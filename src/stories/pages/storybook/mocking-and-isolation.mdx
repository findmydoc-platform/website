# Mocking & isolation

Stories must be isolated: no real navigation, fetch calls, or global app state. Use story-level mocks or decorators to keep story setup consistent and testable.

## Router mocking

Use the shared router decorator when a component expects routing:

```tsx
import { withMockRouter } from '../../utils/routerDecorator'

const meta = {
  title: 'Shared/Templates/Header',
  decorators: [withMockRouter],
}
```

## Fetch mocking

Prefer injecting a deterministic submitter or API client at the component boundary. Keep the story assertion focused on the interaction, not on global network state:

```tsx
const submitClinicRegistration = async () => {
  await new Promise((resolve) => setTimeout(resolve, 120))
  throw new Error('Please review clinic details before submitting.')
}

const meta = {
  title: 'Domain/Auth/Organisms/ClinicRegistrationForm',
  args: {
    onSubmit: submitClinicRegistration,
  },
}
```

## Module mocks

If a component imports a module that should be stubbed for Storybook, register an exact Vite alias in `.storybook/main.ts` and point it to a root `__mocks__/` module. Keep story files free of runner-specific module mocks.

## Rules of thumb

- Prefer story-level decorators over global mocks.
- Keep mocks deterministic and minimal.
- Import Storybook test helpers from `storybook/test`, never from `@storybook/test`.
- Do not import `vitest` directly from story files; use `storybook/test` helpers such as `fn`, `spyOn`, and `mocked`.
- Do not call `vi.mock` or `sb.mock` from story files; imported module mocks belong in `.storybook/main.ts` Vite aliases.
- Do not patch `globalThis.fetch`, `process.env`, or console methods without guaranteed cleanup.
- If a story crashes in tests due to missing providers, fix it in the story, not in global setup.
